# Story 5.3 — Minha Conta: Alterar e-mail e senha

## Status

Done

## Story

**Como** aluno autenticado,
**Quero** poder alterar meu e-mail e minha senha diretamente na página "Minha Conta",
**Para que** eu tenha autonomia completa sobre minhas credenciais de acesso.

## Contexto

Continuação da Story 5.1. Adiciona a seção de segurança de conta à `MinhaContaPage.jsx`, com dois subsistemas independentes: alteração de e-mail e alteração de senha. Ambos usam Supabase Auth diretamente no cliente — sem necessidade de rotas de servidor.

**Métodos Supabase Auth utilizados:**
- `supabase.auth.updateUser({ email: novoEmail })` — envia e-mail de confirmação; a troca só efetiva após o aluno clicar no link
- `supabase.auth.updateUser({ password: novaSenha })` — troca senha imediatamente (não exige senha atual pelo SDK — validar senha atual via re-autenticação: `supabase.auth.signInWithPassword`)
- `supabase.auth.resetPasswordForEmail(email)` — envia link de redefinição para o e-mail atual

## Acceptance Criteria

### AC1 — Seção "Alterar E-mail"
- [ ] Seção "Alterar E-mail" visível na `MinhaContaPage.jsx`
- [ ] Campo de input para o novo e-mail
- [ ] Botão "Atualizar e-mail" chama `supabase.auth.updateUser({ email: novoEmail })`
- [ ] Após chamada bem-sucedida: toast informativo "Um e-mail de confirmação foi enviado para {novoEmail}. O e-mail só será atualizado após a confirmação."
- [ ] O e-mail atual NÃO muda na tela até o usuário confirmar pelo link
- [ ] Toast de erro se a chamada falhar (ex: e-mail já cadastrado)
- [ ] Botão desabilitado se o campo estiver vazio ou igual ao e-mail atual

### AC2 — Seção "Alterar Senha" — Fluxo inline
- [ ] Seção "Alterar Senha" visível com três campos: "Senha atual", "Nova senha", "Confirmar nova senha"
- [ ] Validação client-side: "Nova senha" deve ter no mínimo 8 caracteres
- [ ] Validação client-side: "Nova senha" e "Confirmar nova senha" devem ser iguais
- [ ] Ao clicar "Salvar nova senha":
  1. Re-autentica via `supabase.auth.signInWithPassword({ email, password: senhaAtual })`
  2. Se re-autenticação OK: chama `supabase.auth.updateUser({ password: novaSenha })`
  3. Toast de sucesso "Senha alterada com sucesso"
  4. Limpa os três campos
- [ ] Se re-autenticação falhar: toast de erro "Senha atual incorreta"
- [ ] Se `updateUser` falhar: toast de erro com mensagem da API

### AC3 — Seção "Alterar Senha" — Fluxo por e-mail
- [ ] Botão "Enviar link de redefinição por e-mail" abaixo do formulário inline
- [ ] Ao clicar: chama `supabase.auth.resetPasswordForEmail(user.email)`
- [ ] Toast de sucesso "Link de redefinição enviado para {email}"
- [ ] Botão desabilitado por 60 segundos após o clique (prevenir spam) com contador regressivo visível
- [ ] Toast de erro se a chamada falhar

### AC4 — UX geral
- [ ] As duas seções ("Alterar E-mail" e "Alterar Senha") são independentes — uma não bloqueia a outra
- [ ] Todos os campos sensíveis (senha) usam `type="password"`
- [ ] Botões de submissão ficam desabilitados e exibem "..." durante chamadas assíncronas
- [ ] Nenhum `alert()` ou `confirm()` nativo — tudo via toast
