# Story 1.3 — Deploy de Produção e Validação End-to-End

## Status

Ready

## Story

**Como** administrador da plataforma Mentoria Zero-to-Hero IA,
**Quero** a plataforma funcionando em produção (Vercel + Supabase + AWS S3),
**Para** que os alunos possam acessar o conteúdo e eu possa gerenciar a plataforma sem depender de ambiente local.

## Critérios de Aceite

- [ ] AC-1: Vercel configurado com todas as variáveis de ambiente necessárias e deploy bem-sucedido
- [ ] AC-2: As 4 migrations do Supabase aplicadas em produção (`supabase db push`) sem erros
- [ ] AC-3: Primeiro usuário admin criado no Supabase Auth com `role: admin` em `raw_user_meta_data`
- [ ] AC-4: Login na plataforma (`/plataforma/login`) funciona com email/senha do admin
- [ ] AC-5: Fluxo do aluno validado manualmente: login → módulos → assistir aula (player embarcado) → marcar como concluída → badge ✓ aparece → logout
- [ ] AC-6: Fluxo do admin validado: criar módulo → criar aula com `video_url` → publicar aviso → visualizar progresso
- [ ] AC-7: Upload e download de material via AWS S3 funciona no ambiente de produção
- [ ] AC-8: RLS validada: aluno A não consegue ver progresso do aluno B

## Escopo

### IN (incluso nesta story)
- Configuração de variáveis de ambiente no Vercel (dashboard ou CLI)
- Aplicação das migrations Supabase em produção via `supabase db push`
- Criação do usuário admin via Supabase Dashboard
- Validação manual dos fluxos principais de aluno e admin
- Configuração do bucket S3 com CORS para o domínio de produção Vercel
- Teste de RLS cross-user com dois usuários distintos

### OUT (fora desta story)
- CI/CD automatizado (GitHub Actions para deploy automático)
- Domínio customizado (mentoria.neuralhub.ia.br)
- Monitoramento de erros (Sentry)
- Backup automático do banco

## Dependências

- Stories 1.1 e 1.2 deployadas em `origin/master` ✓
- Conta Vercel com projeto conectado ao repositório `Andre-cardia/Mentoria-IA`
- Projeto Supabase criado com URL e chaves disponíveis
- Bucket AWS S3 criado (`mentoria-ia-materials`) com IAM user configurado
- Supabase CLI instalado localmente (`supabase` command disponível)

## Variáveis de Ambiente Necessárias

### Vercel (Environment Variables)

| Variável | Onde obter |
|----------|-----------|
| `VITE_SUPABASE_URL` | Supabase Dashboard → Project Settings → API → Project URL |
| `VITE_SUPABASE_ANON_KEY` | Supabase Dashboard → Project Settings → API → anon/public key |
| `SUPABASE_SERVICE_KEY` | Supabase Dashboard → Project Settings → API → service_role key |
| `AWS_BUCKET` | Nome do bucket S3 (ex: `mentoria-ia-materials`) |
| `AWS_REGION` | Região do bucket (ex: `us-east-1`) |
| `AWS_ACCESS_KEY_ID` | IAM → Users → mentoria-ia-user → Security credentials |
| `AWS_SECRET_ACCESS_KEY` | Gerado na criação do IAM access key |

> **CRÍTICO:** `VITE_SUPABASE_URL` e `VITE_SUPABASE_ANON_KEY` são variáveis de build (prefixo `VITE_`) — devem ser configuradas como **Environment Variables de Build** no Vercel, não apenas em Runtime.

### AWS S3 — CORS para produção

Adicionar o domínio Vercel na configuração CORS do bucket:

```json
[
  {
    "AllowedHeaders": ["*"],
    "AllowedMethods": ["GET", "PUT"],
    "AllowedOrigins": [
      "http://localhost:5173",
      "https://*.vercel.app",
      "https://mentoria.neuralhub.ia.br"
    ],
    "ExposeHeaders": ["ETag"]
  }
]
```

## Riscos

| Risco | Probabilidade | Impacto | Mitigação |
|-------|--------------|---------|-----------|
| Migrations falham por dados inconsistentes no banco de produção | Baixa | Alto | Executar em banco limpo; fazer backup antes de `db push` |
| CORS do S3 não configurado para domínio Vercel | Alta | Alto | Atualizar CORS antes de testar upload em produção |
| Variáveis `VITE_*` não aplicadas no build (apenas em runtime) | Média | Alto | Confirmar que Vercel aplica estas vars em tempo de build no painel |
| `supabase db push` requer projeto vinculado ao CLI | Média | Médio | Executar `supabase login` e `supabase link --project-ref <ref>` antes |

## Dev Notes

### Obter Project Ref do Supabase

```bash
# No dashboard Supabase: Settings → General → Reference ID
# Exemplo: abcdefghijklmnop
supabase link --project-ref abcdefghijklmnop
```

### Verificar migrations aplicadas

```bash
supabase migration list
# Deve listar as 4 migrations com status "applied"
```

### Criar admin via SQL (alternativa ao dashboard)

```sql
-- Executar no Supabase SQL Editor
UPDATE auth.users
SET raw_user_meta_data = raw_user_meta_data || '{"role": "admin"}'::jsonb
WHERE email = 'seu-email@exemplo.com';
```
